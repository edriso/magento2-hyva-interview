What is Magento 2?
The big picture

Magento 2 is a powerful, open-source e-commerce platform built with PHP. You install it on a server and it gives you everything — product pages, cart, checkout, admin panel. You customize it by adding your own code in modules.

Analogy: Magento is a giant LEGO set. The base kit is a working store. You add your own pieces (modules) on top without breaking the original.

Why Magento over Shopify?

ReasonExplanation
Full server controlYou own everything — no transaction fees, no platform rules, no monthly caps
Native B2B featuresCompany accounts, custom pricing, approval workflows — built-in, not expensive apps
Unlimited customizationYou can change literally anything with PHP code
Large complex catalogsBetter for complex pricing rules, multiple stores, multi-language

Key Facts

  • Written in PHP, uses MySQL database
  • Adobe owns it — Adobe Commerce = paid, Magento Open Source = free
  • All code organized into modules (folders with specific structure)
  • The complexity is a feature — it enables deep customization

Module Structure
How code is organized

All Magento code lives in modules. A module is a folder. Each module does one thing. You write your custom code the same way Magento does.

Analogy: A module is like a WordPress plugin — you can enable/disable it, and it doesn't break other things.

The 2 Required Files

FilePurpose
registration.phpTells Magento "I exist" — without this, the module is ignored
etc/module.xmlDeclares the module name and version
// registration.php
\Magento\Framework\Component\ComponentRegistrar::register(
    \Magento\Framework\Component\ComponentRegistrar::MODULE,
    'Vendor_ModuleName',
    __DIR__
);
<!-- etc/module.xml -->
<config>
    <module name="Vendor_ModuleName" setup_version="1.0.0"/>
</config>

Full Folder Structure

app/code/Vendor/ModuleName/
├── registration.php       ← required
├── etc/
│   ├── module.xml         ← required
│   ├── di.xml             ← dependency injection
│   └── events.xml         ← event listeners
├── Plugin/                ← interceptors
├── Observer/              ← event observers
├── ViewModel/             ← PHP logic for templates
├── Setup/Patch/Data/      ← database patches
└── view/frontend/
    ├── layout/            ← page structure XML
    ├── templates/         ← .phtml files
    └── web/               ← JS, CSS, images

After Creating a Module

bin/magento module:enable Vendor_ModuleName
bin/magento setup:upgrade
bin/magento cache:flush

MVC Architecture
How a request flows through Magento
Analogy: Think of a restaurant. Controller = waiter (takes order). Model = kitchen (gets/saves data). View = the plate of food (what user sees).

Full Request Flow

Browser request
    → index.php (entry point)
    → Router (finds Controller based on URL)
    → Controller (handles request logic)
    → ViewModel / Block (prepares data for template)
    → Layout XML (decides which template to load)
    → Template .phtml (outputs HTML)
    → Browser receives HTML

URL Pattern

domain.com / module / controller / action

URLModuleControllerAction
/catalog/product/viewcatalogproductview
/checkout/cart/indexcheckoutcartindex
/customer/account/logincustomeraccountlogin

Block vs ViewModel

BlockViewModel
StatusOld way (still used)New, preferred way
PurposePass data to templatesPass data to templates
UseAvoid for new codeUse this for new code

Layout XML
Page blueprints — what goes where

Layout XML files are blueprints that tell Magento which blocks/templates to show on which pages. The filename = the page it applies to.

Analogy: Layout XML is like a floor plan. It doesn't contain furniture (content), it says where rooms (blocks) go and what's inside each room.

Key File Names

FileApplies to
default.xmlALL pages
catalog_product_view.xmlProduct pages only
catalog_category_view.xmlCategory pages only
checkout_cart_index.xmlCart page only
cms_index_index.xmlHomepage only

Key Directives

<!-- Add a block to the content area -->
<referenceContainer name="content">
    <block class="Vendor\Module\ViewModel\MyBlock"
           name="my.block"
           template="Vendor_Module::my-template.phtml" />
</referenceContainer>

<!-- Remove a block from this page -->
<referenceBlock name="some.block.name">
    <action method="setTemplate"><argument name="template" xsi:type="string"></argument></action>
</referenceBlock>

<!-- Move a block to a different container -->
<move element="block.name" destination="container.name" />

Rules

  • Never edit core layout files — create your own in your theme or module
  • File placed at: view/frontend/layout/
  • This is how you add/remove content without touching core PHP

Plugins (Interceptors)
Change any method without editing core

A plugin lets you add code before, after, or around any public method in Magento — without editing the original class. This is the #1 way to customize Magento safely.

Analogy: A plugin is like a security guard at a door. Before someone enters (before), you can check them. After they leave (after), you can log it. You can even stop them (around).

Three Types

TypeWhen it runsWhat you can do
beforeBefore the original methodChange the input arguments
afterAfter the original methodChange the return value
aroundWraps the originalFull control — can even skip original

1. Register in di.xml

<type name="Magento\Catalog\Model\Product">
    <plugin name="my_product_plugin"
            type="Vendor\Module\Plugin\ProductPlugin" />
</type>

2. The Plugin Class

class ProductPlugin
{
    // Runs AFTER Product::getName() — modifies the return value
    public function afterGetName(
        \Magento\Catalog\Model\Product $subject,
        $result  // original return value
    ) {
        return $result . ' (Sale!)';
    }
}

Rules

  • Only works on public methods
  • Method naming: beforeMethodName / afterMethodName / aroundMethodName
  • Prefer before / after over around — around is powerful but slower
  • Registered in etc/di.xml

Events & Observers
React to things that happen

Magento fires events when things happen (order placed, product saved, etc.). You write an Observer to listen for that event and run your own code.

Analogy: Events are like a fire alarm. When it fires, the sprinklers (observers) react. The alarm doesn't know about the sprinklers — they just listen and respond.

Common Events

Event nameWhen it fires
checkout_submit_all_afterOrder placed
catalog_product_save_afterProduct saved in admin
customer_loginCustomer logged in
sales_order_place_afterOrder placed (alternative)

1. Register in events.xml

<event name="checkout_submit_all_after">
    <observer name="my_order_observer"
              instance="Vendor\Module\Observer\OrderObserver" />
</event>

2. The Observer Class

class OrderObserver implements ObserverInterface
{
    public function execute(Observer $observer)
    {
        $order = $observer->getData('order');
        // send email, update external system, etc.
    }
}

Plugin vs Observer — When to Use Which

PluginObserver
Use whenYou need to change a method's input or outputYou need to react to something that happened
Works onAny public methodOnly Magento-fired events
ExampleModify product name before displaySend email when order is placed

Dependency Injection
How classes get the tools they need

Instead of creating objects yourself inside a class, Magento injects (passes) them to you through the constructor. Magento reads your constructor and automatically gives you what you asked for.

Analogy: You need a pen to write. Bad way: you make the pen yourself every time. Good way: someone hands you a pen when you sit down. DI is "someone hands you the pen."

Bad Way vs Good Way

// ❌ BAD — creating manually
class MyClass {
    public function doSomething() {
        $product = new \Magento\Catalog\Model\Product();
    }
}

// ✅ GOOD — dependency injection
class MyClass {
    private $productRepository;

    public function __construct(
        \Magento\Catalog\Api\ProductRepositoryInterface $productRepository
    ) {
        $this->productRepository = $productRepository;
    }
}

Why Inject Interfaces, Not Classes?

  • You can swap the implementation later without changing your class
  • Easier to test (you can inject a mock)
  • Follows the Magento contract (API guarantee)

Preference in di.xml

When someone asks for an interface, tell Magento which class to use:

<preference for="Magento\Catalog\Api\ProductRepositoryInterface"
            type="Magento\Catalog\Model\ProductRepository" />
Interview tip: "I always inject interfaces, not concrete classes. This makes the code flexible — if Magento or a module overrides the implementation, my code automatically gets the new version."

Data Patches
One-time database changes

A Data Patch is a PHP class that runs once during setup:upgrade to make changes to the database — add a CMS block, set a config value, create an attribute.

Analogy: A Data Patch is like a one-time checklist task. Once it's done, it's marked complete and never runs again — even if you run setup:upgrade 100 times.

Structure

namespace Vendor\Module\Setup\Patch\Data;

class AddWelcomeBanner implements DataPatchInterface
{
    public function apply()
    {
        // Your one-time DB change here
        // e.g., create a CMS block, set a config value
    }

    public static function getDependencies(): array
    {
        return []; // Other patches that must run first
    }

    public function getAliases(): array
    {
        return [];
    }
}

Rules

RuleExplanation
Place in Setup/Patch/Data/Magento auto-discovers it here
Tracked in patch_list DB tableWon't run twice
Never modify a deployed patchCreate a new patch instead

Schema vs Data Patches

TypeUsed for
Schema PatchChange DB structure (add/remove columns, tables)
Data PatchChange DB content (insert/update data, CMS blocks, config)

EAV Model
How Magento stores product attributes

EAV = Entity-Attribute-Value. Instead of one giant table with hundreds of columns, Magento splits product data across many small tables. Each row stores one attribute value for one entity.

Analogy: Normal DB = one big spreadsheet with a column per property. EAV = a notebook where each page says "Product #5, attribute: color, value: red." Very flexible, but more complex to query.

The 3 Concepts

ConceptExampleDB Table
EntityA product, a customer, a categorycatalog_product_entity
Attribute"color", "size", "price"eav_attribute
Value"red", "XL", "29.99"catalog_product_entity_varchar (etc.)

Why Magento Uses EAV

  • You can add custom product attributes without changing the DB schema
  • Products can have different sets of attributes (shirt has "size", TV has "screen size")
  • Very flexible for complex catalogs

The Trade-off

EAV makes Magento flexible but complex. Fetching a product requires joining many tables, which is slower than a single table. That's why Magento also uses "flat tables" for faster frontend reads — a pre-joined snapshot.

What is Hyvä?
The modern Magento frontend

Hyvä is a replacement for the default Magento frontend theme (Luma). It removes old, heavy technology and replaces it with fast, modern tools.

Analogy: Luma is a diesel truck — powerful but heavy and slow to start. Hyvä is a sports car — lighter, faster, modern.

Alpine.js Basics
10 directives you must know

Alpine.js is a tiny JS library used in Hyvä. You write it directly in HTML using special attributes called directives.

Analogy: Alpine.js is like sticky notes on HTML elements. "Hey button — when clicked, show this div."

The 10 Directives

DirectiveWhat it doesExample
x-dataDeclare reactive statex-data="{ open: false }"
x-showShow/hide (stays in DOM)x-show="open"
x-ifShow/hide (removes from DOM)<template x-if="open">
x-textSet text contentx-text="count"
x-modelTwo-way input bindingx-model="email"
@clickClick event listener@click="open = true"
x-forLoop over arrayx-for="item in items"
:classDynamic class binding:class="{ active: open }"
$dispatchFire custom event$dispatch('add-to-cart', {id:1})
$storeAccess global state$store.cart.count

x-show vs x-if

x-showx-if
DOMElement stays, hidden with CSSElement removed/added from DOM
Use whenToggled frequently (dropdown, modal)Shown rarely (error messages)
PerformanceFaster to toggleFaster when hidden most of the time

Reusable Components with Alpine.data()

// In a .js file
Alpine.data('storeSelector', () => ({
    stores: [],
    selected: null,
    init() { /* runs automatically on load */ },
    selectStore(id) { this.selected = id; }
}));

// In HTML template
<div x-data="storeSelector()">...</div>

TailwindCSS in Hyvä
Styling directly in HTML

TailwindCSS is a CSS framework where you style elements directly in HTML using small utility classes. No separate CSS files needed for most things.

Analogy: Normal CSS: write class .big-red-button in a separate file. Tailwind: write class="bg-red-500 text-white px-4 py-2 rounded" directly on the element.

Example

<!-- Without Tailwind -->
<button class="my-button">Buy Now</button>
<!-- needs separate CSS file for .my-button -->

<!-- With Tailwind -->
<button class="bg-blue-600 text-white px-6 py-3 rounded-lg hover:bg-blue-700">
    Buy Now
</button>
<!-- all styling is right here -->

Common Utility Classes to Know

CategoryExamples
Layoutflex, grid, block, hidden
Spacingmt-4, px-6, py-3, gap-4
Typographytext-lg, font-bold, text-gray-600
Colorsbg-blue-600, text-white, border-gray-200
Hoverhover:bg-blue-700, hover:text-white

Key Facts

  • Config file location: app/design/frontend/Vendor/Theme/web/tailwind/tailwind.config.js
  • Tailwind only includes CSS classes you actually use → very small final file
  • Run npm run build in the tailwind folder to compile
  • Result: a single css/styles.css file served by Magento

Security
OWASP, PCI DSS, and Magento hardening

OWASP Top Threats

AttackWhat it isHow Magento handles it
SQL InjectionAttacker puts SQL in a form to read/delete your DBMagento uses ORM — never raw SQL
XSSAttacker injects JavaScript into your pageMagento escapes all output with escapeHtml()
CSRFTricks browser into making requests user didn't intendMagento uses form keys (tokens) on all forms
Broken AuthWeak passwords, no 2FAEnforce strong passwords + 2FA in admin
Data ExposureSensitive data not encryptedHTTPS always, never store card numbers

Magento Security Checklist

  • Change /admin URL to something secret
  • Enable 2FA for admin users
  • Restrict admin access by IP
  • HTTPS only (redirect HTTP)
  • Correct file permissions (664/775)
  • Hide Magento version from public
  • WAF rules in Cloudflare
  • Regular DB + media backups
  • Subscribe to security patches
  • Use hosted payment gateways

PCI DSS (Payment Cards)

Rules you must follow if you handle credit cards. Easiest approach: use a hosted gateway (Stripe, Paymob) — they handle card data, not you. You never store card numbers.

GDPR (EU Data Privacy)

  • Cookie consent banner required
  • Users can request account data deletion
  • Email marketing must be opt-in
