Magento 2 Core Concepts
Architecture, modules, how everything fits together

🏗️ What IS Magento 2? (The Big Picture)

Magento 2 is a PHP-based e-commerce platform. Think of it like a huge LEGO set — it comes with hundreds of ready-made pieces (modules), and you can add/override/build new pieces without touching the core.

REQUEST → index.php → Bootstrap → Routing → Controller ↓ Block / ViewModel (PHP logic) ↓ Layout XML (decides what to show) ↓ Template .phtml (HTML output)

Key principle: Never modify core files. Always extend via modules, plugins, or theme overrides. This way upgrades don't break your work.

📦 Module Structure — The Building Block

Every feature in Magento lives in a module. Here's the minimum structure:

app/code/Vendor/ModuleName/
├── registration.php          ← tells Magento "I exist"
├── etc/
│   └── module.xml            ← module name + dependencies
├── etc/frontend/
│   └── di.xml                ← dependency injection config
├── Plugin/                   ← intercept other classes
├── Observer/                 ← listen to events
├── ViewModel/                ← PHP logic for templates
└── view/frontend/
    ├── layout/               ← layout XML files
    └── templates/            ← .phtml template files

After creating a module: Run bin/magento setup:upgrade to register it, then bin/magento cache:flush

🎛️ Layout XML — Magento's "Page Blueprint"

Layout XML files decide WHAT appears on a page and WHERE. They're XML files that add/remove/move blocks.

<!-- catalog_product_view.xml — runs on product pages -->
<page>
  <body>
    <referenceContainer name="content">
      <block class="MyBlock" name="my.block" template="template.phtml">
        <arguments>
          <argument name="view_model" xsi:type="object">My\ViewModel</argument>
        </arguments>
      </block>
    </referenceContainer>
  </body>
</page>

Common layout handles:

  • default.xml — every page
  • catalog_product_view.xml — product page
  • catalog_category_view.xml — category page
  • checkout_cart_index.xml — cart
  • customer_account_*.xml — account pages

Key actions:

  • referenceBlock — modify existing block
  • referenceContainer — modify container
  • remove — hide a block
  • move — move block to another container

🔌 Plugins — Intercepting Any Method

A Plugin lets you run code BEFORE, AFTER, or AROUND any public method in Magento — without editing the original class.

// di.xml — register the plugin
<type name="Magento\Catalog\Model\Product">
  <plugin name="my_plugin" type="Vendor\Module\Plugin\ProductPlugin"/>
</type>

// ProductPlugin.php
public function afterGetName(\Magento\Catalog\Model\Product $subject, string $result): string {
    return $result . ' — Sale!';  // modifies the return value
}
TypeWhen it runsCan change?
beforeBefore the original methodInput arguments
afterAfter the original methodReturn value
aroundWraps the original methodEverything (use sparingly)

📣 Events & Observers — React to Things That Happen

Magento fires events at key moments (checkout completes, product saves, etc.). You can listen and react.

// events.xml
<event name="checkout_submit_all_after">
  <observer name="my_observer" instance="Vendor\Module\Observer\OrderObserver"/>
</event>

// OrderObserver.php
public function execute(\Magento\Framework\Event\Observer $observer): void {
    $order = $observer->getEvent()->getOrder();
    // do something with the order
}

Plugin vs Observer: Use observers when Magento fires an event at exactly the right moment. Use plugins when you need to intercept a specific method that doesn't have an event.

💉 Dependency Injection (DI) — How Classes Get Their Dependencies

In Magento, you never use new ClassName(). Instead, you declare what you need in the constructor, and Magento injects it automatically. This is called DI.

class MyViewModel implements ArgumentInterface
{
    public function __construct(
        private readonly ProductRepositoryInterface $productRepository,
        private readonly StoreManagerInterface $storeManager
    ) {}

    public function getProduct(int $id): ProductInterface {
        return $this->productRepository->getById($id);
    }
}

The di.xml file lets you configure which implementation to inject (e.g., swap a class, add a plugin, configure preferences).

🗃️ Data Patches — Installing Data via Code

When you need to insert/update database data as part of your module (CMS blocks, config values, attributes), use a Data Patch. It runs once via bin/magento setup:upgrade.

class AddMyBlock implements DataPatchInterface
{
    public function apply(): void {
        // create a CMS block, insert config, create attribute, etc.
        $this->moduleDataSetup->startSetup();
        // ... your code
        $this->moduleDataSetup->endSetup();
    }

    public static function getDependencies(): array { return []; }
    public function getAliases(): array { return []; }
}

Never modify a deployed patch — create a new one that depends on the original instead. Editing a deployed patch means everyone must manually delete it from the DB and re-run.

🧩 EAV Model — The Product Attribute System

EAV = Entity-Attribute-Value. It's how Magento stores product data so flexibly. Instead of adding columns to a table, attributes are stored as separate rows.

Normal DB: products table → id | name | price | color | size | ... EAV in Magento: catalog_product_entity ← the "entity" (product) catalog_product_entity_varchar ← text values catalog_product_entity_decimal ← price/number values catalog_product_entity_int ← yes/no, selects

Why? Allows unlimited custom attributes without schema changes. Downside? Slower queries (joins across many tables).

🤔 What is Hyvä and Why Does It Exist?

The default Magento frontend (Luma) uses RequireJS + KnockoutJS — both old, slow, and hard to work with. Hyvä replaces ALL of that with a much simpler, faster stack.

Luma (default)Hyvä
RequireJS (module loader)❌ Gone — plain JS or Alpine.js
KnockoutJS (data binding)❌ Gone — Alpine.js instead
Bootstrap 3 CSS❌ Gone — TailwindCSS instead
Page load: ~3-5 seconds✅ Page load: <1 second (often)
JS bundle: 800KB+✅ JS bundle: ~40KB

Performance numbers you can quote: Hyvä typically improves Google Lighthouse score from 30-50 to 90+. That's a 3x improvement that directly impacts SEO and conversion rates.

🏔️ Alpine.js — The JavaScript of Hyvä

Alpine.js is like a lightweight Vue/React but lives in your HTML. No build step, no component files — just x- attributes on your HTML tags.

<!-- Simple counter example -->
<div x-data="{ count: 0 }">
  <button @click="count++">+</button>
  <span x-text="count"></span>
</div>

<!-- Real Magento example: toggle mobile menu -->
<div x-data="{ isOpen: false }">
  <button @click="isOpen = !isOpen">Menu</button>
  <nav x-show="isOpen" x-transition>...</nav>
</div>

Key directives:

  • x-data — declare reactive data
  • x-show — show/hide element
  • x-text — set text content
  • x-bind or : — bind attribute
  • x-on or @ — event listener
  • x-model — two-way form binding
  • x-if — conditional rendering
  • x-for — loop rendering

Advanced features used in projects:

  • $dispatch('event-name', data) — fire custom event
  • $store.varName — global state store
  • Alpine.data('name', () => {...}) — reusable component
  • x-init — run JS on init
  • x-cloak — hide until Alpine loads (prevents flash)
  • x-transition — CSS transitions

🎨 TailwindCSS in Hyvä

Instead of writing custom CSS files, you style with utility classes directly in your HTML. Hyvä runs a Tailwind build step that scans your templates and produces a tiny CSS file.

<!-- Before (Luma style) -->
<div class="product-card"> ... </div>
/* CSS file: .product-card { display: flex; padding: 16px; ... } */

<!-- Hyvä style -->
<div class="flex flex-col p-4 bg-white rounded-lg shadow hover:shadow-md"> ... </div>

Config location: app/design/frontend/Vendor/Theme/web/tailwind/tailwind.config.js — this is where you define project colors, fonts, and custom values. Always check this first on a new project.

🔄 How Template Override Works in Hyvä

Hyvä overrides Magento core templates by placing files in the theme with the same path. Magento's fallback chain looks here first.

Magento looks for templates in this order: 1. Your theme: app/design/frontend/Vendor/Theme/ 2. Parent theme: vendor/hyva-themes/magento2-default-theme/ 3. Module: vendor/magento/module-catalog/view/frontend/ Example — override product name template: Core path: Magento_Catalog/templates/product/view/default.phtml Your file: app/design/frontend/Vendor/Theme/ Magento_Catalog/templates/product/view/default.phtml

Don't copy-paste full templates if you only need small changes — use Layout XML + a new block with a partial template instead. Full overrides break on Magento upgrades.

How do you customize Magento without touching core files?
TECHNICAL ▼
The three ways

How would you integrate a payment gateway in Magento?
TECHNICAL ▼
The approach

Magento has a standard payment gateway interface. You either:

  1. Use an existing module — most gateways (Stripe, PayPal, Braintree, Paymob for Egypt) have official Magento modules. You install via Composer and configure in Admin > Stores > Configuration.
  2. Build a custom gateway — implement PaymentInterface, handle authorize/capture/refund methods, add it to checkout config.

The key files are etc/config.xml (default config), etc/payment.xml (payment method config), and the Gateway Command classes that handle the actual API calls.

For Egypt/MENA: Paymob, Fawry, and MyFatoorah all have Magento 2 modules available.

How would you handle a Magento upgrade or security patch?
TECHNICAL ▼
The process
  1. Test on staging first — never on production directly
  2. Update via Composer: composer update magento/product-community-edition --with-all-dependencies
  3. Run bin/magento setup:upgrade and setup:di:compile
  4. Deploy static content: setup:static-content:deploy
  5. Check for conflicts — any templates you overrode might need updates
  6. Run full QA on staging
  7. Deploy to production during low-traffic window with a rollback plan

Why proper module structure matters: Because you never touch core files, upgrades only break things if a template you overrode changed in the core. This is why you minimize full template overrides.

How would you handle security in Magento? (PCI DSS, OWASP)
INFRA ▼
Key points

OWASP Top 10 in Magento context:

  • SQL Injection — Magento's ORM (collections, repositories) parameterizes queries. Never concatenate user input into SQL.
  • XSS — Always use escapeHtml(), escapeUrl() in templates. Magento blocks SVG uploads by default (XSS vector).
  • CSRF — Magento adds CSRF tokens to all forms automatically.
  • Auth — Enforce strong admin passwords, change admin URL from /admin to custom path, enable 2FA.

PCI DSS basics: If handling card payments, use a hosted payment gateway (Stripe, etc.) so card data never touches your server — this takes you out of PCI scope.

Regular maintenance: Apply Magento security patches promptly (Adobe releases them monthly). Subscribe to Magento security alerts.

Cloudflare (WAF + CDN + DDoS) ↓ GCP Load Balancer ↓ Compute Engine VM (Ubuntu/Debian) ├── Nginx (web server, handles HTTP) ├── PHP-FPM 8.1+ (runs Magento PHP code) ├── Redis (caching + sessions) └── Elasticsearch/OpenSearch (search indexing) ↓ Cloud SQL (MySQL 8.0 — separate managed instance) ↓ Cloud Storage (product images, media files)

🐧 Linux Commands You Should Know

# File permissions (Magento needs these)
find var/ -type f -exec chmod 664 {} \;
find var/ -type d -exec chmod 775 {} \;
chown -R www-data:www-data .

# Check what's running
systemctl status nginx
systemctl status php8.1-fpm

# View logs
tail -f /var/log/nginx/error.log
tail -f var/log/exception.log
# Nginx reload (after config change)
nginx -t && systemctl reload nginx

# Check disk space
df -h

# Check memory
free -h

# Check running processes
htop  # or top

# MySQL
mysql -u root -p
USE magento_db;

🔒 Security Checklist for a New Magento Install

  • ✅ Change admin URL from /admin to something custom (e.g., /backend-xyz)
  • ✅ Enable 2FA for admin panel
  • ✅ Strong admin passwords (16+ chars)
  • ✅ Restrict admin access to specific IPs via Cloudflare/Nginx
  • ✅ HTTPS only — redirect all HTTP to HTTPS
  • ✅ Set proper file permissions (664 for files, 775 for dirs)
  • ✅ Hide Magento version from headers
  • ✅ WAF rules in Cloudflare (block common Magento exploit patterns)
  • ✅ Regular backups — database + media files
  • ✅ Subscribe to Adobe security alerts — apply patches quickly
  • ✅ Never store card numbers — use hosted payment gateways
  • ✅ GDPR: cookie consent, data deletion capability, privacy policy

🗄️ Backup Strategy (What to Mention)

For a production Magento store, you need to back up:

  • Database (MySQL) — most important. Daily automated backups via Cloud SQL, keep 30 days. Also before any deployment.
  • Media files (pub/media/) — product images etc. Store on Cloud Storage with versioning.
  • Code — in Git. Full history is your code backup.

Disaster recovery: Test your backups by actually restoring them on a staging environment. A backup you've never tested is not a backup.
