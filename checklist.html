Magento 2 architecture overview (module, layout XML, MVC)

How the whole thing is structured — the mental model you need

▼

The Big Picture — MVC in Magento

MVC = Model, View, Controller. A browser request hits Magento and travels through this chain:

Browser → index.php → Bootstrap → Router → Controller → (Model + ViewModel) → View (Layout + Template) ↓ Sends HTML back to browser

Each Layer Explained Simply

LayerWhat it doesWhere it lives
ControllerReceives the request, decides what to show, never contains business logicController/
ModelDatabase interaction — reads and writes dataModel/
ViewModelPHP logic that prepares data for templates (the Magento-modern way)ViewModel/
Layout XMLDecides which blocks appear on which pages and in what orderview/frontend/layout/
Template (.phtml)The actual HTML with small PHP variables sprinkled inview/frontend/templates/

Module — The Unit of Everything

Magento is entirely built from modules. Every feature (catalog, checkout, customer, search) is its own module. Your custom code lives in modules too.

app/code/Vendor/ModuleName/
├── registration.php        ← tells Magento "this module exists"
├── etc/
│   ├── module.xml          ← module name, version, dependencies
│   ├── di.xml              ← dependency injection config (plugins, preferences)
│   └── frontend/
│       ├── di.xml          ← DI config for frontend area
│       └── routes.xml      ← registers URL routes
├── Controller/Frontend/
│   └── Index.php           ← handles GET /myroute/index/index
├── ViewModel/
│   └── MyData.php          ← PHP logic for templates
├── Model/
│   └── MyModel.php         ← data/business logic
└── view/frontend/
    ├── layout/
    │   └── myroute_index_index.xml   ← layout for that URL
    └── templates/
        └── myblock.phtml   ← the HTML template

Layout XML — The Blueprint of Each Page

Layout XML files answer: "Which blocks appear on this page, in what container, in what order?" They're XML files that add/remove/configure blocks.

<page xmlns:xsi="..." xsi:noNamespaceSchemaLocation="...">
  <body>
    <referenceContainer name="content">
      <block class="Magento\Framework\View\Element\Template"
             name="my.block"
             template="Vendor_Module::myblock.phtml">
        <arguments>
          <argument name="view_model" xsi:type="object">Vendor\Module\ViewModel\MyData</argument>
        </arguments>
      </block>
    </referenceContainer>
  </body>
</page>

File name = page handle — the filename determines which page it runs on:

  • default.xml → every page
  • catalog_product_view.xml → product detail page
  • catalog_category_view.xml → category / PLP
  • customer_account_index.xml → customer dashboard
  • checkout_cart_index.xml → cart page
Say in interview

"Magento follows MVC — controllers handle routing, models handle data, and ViewModels prepare data for templates. What's unique to Magento is the Layout XML system — XML files that act as a blueprint for each page, declaring which blocks appear where. This separation means the same block can be added or removed on any page just by changing an XML file, without touching PHP."

Plugins (before/after/around) — know how to explain with an example

The most important Magento customization tool

▼

What Is a Plugin?

A Plugin (also called an Interceptor) lets you run your own code before, after, or around any public method in Magento — without modifying the original class.

Think of it like wrapping a function. The original function still exists and runs normally — you just get to run code before it, after it, or completely wrap it.

Step 1 — Register the Plugin in di.xml

<!-- app/code/Vendor/Module/etc/di.xml -->
<type name="Magento\Catalog\Model\Product">
  <plugin name="vendor_module_product_plugin"
          type="Vendor\Module\Plugin\ProductPlugin"
          sortOrder="10" />
</type>

Step 2 — Write the Plugin Class

namespace Vendor\Module\Plugin;

class ProductPlugin
{
    // BEFORE — runs before getPrice(), can modify the arguments
    public function beforeGetPrice(\Magento\Catalog\Model\Product $subject): array
    {
        // return modified args as array (even if no args, return [])
        return [];
    }

    // AFTER — runs after getPrice(), can modify the return value
    public function afterGetPrice(\Magento\Catalog\Model\Product $subject, float $result): float
    {
        // $result = what getPrice() returned
        return $result * 0.9; // 10% discount on everything
    }

    // AROUND — wraps the method entirely
    public function aroundGetPrice(\Magento\Catalog\Model\Product $subject, callable $proceed): float
    {
        // $proceed() calls the original method
        if ($subject->getSku() === 'SPECIAL') {
            return 0.0; // free item, don't call original
        }
        return $proceed(); // normal flow
    }
}
Plugin typeMethod prefixWhat you can doWhen to use
beforebeforeMethodNameModify input argumentsValidate or transform inputs
afterafterMethodNameModify the return valueAdd to results, format output
aroundaroundMethodNameControl everything including whether the original runsRarely — only when before/after aren't enough

Avoid around plugins unless necessary. They're hard to debug, slow performance if overused, and prevent other plugins from stacking cleanly. Always prefer before/after.

Say in interview

"A plugin lets me intercept any public method in Magento without touching the original class. For example, if I want to add a 10% discount to product prices for a specific customer group, I write an after-plugin on the getPrice() method. The original method still runs — I just get to modify what it returns. This keeps the core untouched so upgrades never break my customizations."

Events & Observers — difference from plugins

React to things that happen in Magento

▼

The Analogy

Think of events like notifications. Magento says "I just did something" — and anyone who signed up to listen gets notified. You write an Observer to be that listener.

How It Works — 3 Files

1. events.xml — subscribe to the event
<config>
  <event name="checkout_submit_all_after">
    <observer name="vendor_send_confirmation"
              instance="Vendor\Module\Observer\SendConfirmation"/>
  </event>
</config>

2. Observer/SendConfirmation.php — your logic
class SendConfirmation implements ObserverInterface
{
    public function execute(Observer $observer): void
    {
        $order = $observer->getEvent()->getOrder();
        // send email, call API, log something, etc.
    }
}

3. That's it. Run bin/magento cache:flush and it works.

Common Events You Should Know

Event nameWhen it fires
checkout_submit_all_afterAfter order is placed
catalog_product_save_afterAfter a product is saved in admin
customer_loginAfter customer logs in
customer_register_successAfter new customer registers
sales_order_place_afterAfter order is placed (similar to above)
controller_front_send_response_beforeBefore response is sent (every request)

Plugin vs Observer — When to Use Which

PluginObserver
Use whenYou need to intercept a specific method — change its inputs or outputsMagento already fires an event at the exact moment you need
Can modify return?✅ Yes❌ No — just react
Needs specific trigger?Works on any public methodMust be an event Magento fires
Best forChanging how something worksReacting after something happened

Rule of thumb: If Magento fires an event at the exact right moment → use an observer. If not, or if you need to change a return value → use a plugin.

Say in interview

"Events and Observers are Magento's pub/sub system. Magento fires events at key moments — like after an order is placed — and I write an Observer to react to that event without touching the original checkout code. The difference from plugins is that plugins intercept a method and can modify its output; observers just react to a broadcast and can't change what already happened."

Dependency Injection — what it is, why Magento uses it

The pattern that makes Magento's code testable and flexible

▼

The Problem DI Solves

Without DI, you'd write: $repo = new ProductRepository();. This hard-codes the dependency — you can't swap it, test with a mock, or configure it differently per environment.

With DI, you declare what you need in the constructor, and Magento creates and injects it for you.

Without DI (bad):

class MyClass {
  public function getData() {
    // hard-coded, can't swap it
    $repo = new ProductRepository();
    return $repo->getList(...);
  }
}

With DI (Magento way):

class MyClass {
  public function __construct(
    private readonly ProductRepositoryInterface $repo
  ) {}

  public function getData() {
    return $this->repo->getList(...);
  }
}

How Magento Knows What to Inject

Magento reads di.xml files and its own type definitions. If you type-hint an interface, Magento looks up which concrete class implements it (set in di.xml via <preference>).

<!-- di.xml — tell Magento: when someone asks for the interface, give them this class -->
<preference for="Magento\Catalog\Api\ProductRepositoryInterface"
            type="Magento\Catalog\Model\ProductRepository"/>

Why This Matters

  • Testable — in unit tests you inject a mock instead of the real class
  • Swappable — want to replace Magento's search with Algolia? Register a preference in di.xml, no code changes needed
  • Configurable — different implementations per area (frontend vs adminhtml)
  • Single Responsibility — each class has one job; dependencies are provided externally

Always inject interfaces, not concrete classes. ProductRepositoryInterface not ProductRepository. This is the Magento convention and it's enforced in code reviews.

Say in interview

"Dependency Injection means a class declares what it needs in its constructor, and Magento's Object Manager provides those dependencies automatically. Instead of writing 'new ProductRepository()' inside my class, I type-hint the interface in the constructor and Magento injects the right implementation. This makes code testable, loosely coupled, and easy to extend — which is why all of Magento is built this way."

Data Patches — when to use, gotchas (never modify deployed)

How to install database data as part of your module

▼

What Is a Data Patch?

A Data Patch is a PHP class that runs once when you run bin/magento setup:upgrade. After it runs, Magento records it in the patch_list table so it never runs again.

Use it for: creating CMS blocks/pages, inserting config values, adding product attributes, seeding data your module needs.

Basic Structure

namespace Vendor\Module\Setup\Patch\Data;

use Magento\Framework\Setup\Patch\DataPatchInterface;
use Magento\Framework\Setup\ModuleDataSetupInterface;

class AddWelcomeBlock implements DataPatchInterface
{
    public function __construct(
        private readonly ModuleDataSetupInterface $moduleDataSetup,
        private readonly \Magento\Cms\Model\BlockFactory $blockFactory
    ) {}

    public function apply(): void
    {
        $this->moduleDataSetup->startSetup();

        $this->blockFactory->create()->setData([
            'identifier' => 'welcome-banner',
            'title'      => 'Welcome Banner',
            'content'    => '<p>Welcome!</p>',
            'is_active'  => 1,
        ])->save();

        $this->moduleDataSetup->endSetup();
    }

    // If this patch depends on another patch running first:
    public static function getDependencies(): array
    {
        return [SomeOtherPatch::class];
    }

    public function getAliases(): array { return []; }
}

The One Rule You Must Know

NEVER modify a patch that has already been deployed to production. Once it's in patch_list, Magento won't run it again. If you edit it, only developers who delete it from the DB manually will see the change. Everyone else gets nothing — silently broken data.

The right way: Create a new patch that depends on the original:

// NEW patch — don't touch the original
class UpdateWelcomeBlock implements DataPatchInterface
{
    public static function getDependencies(): array
    {
        return [AddWelcomeBlock::class]; // runs after the first one
    }
    // update the block here
}

Other Patch Types

  • Data Patch — inserts/modifies database records (what you use 99% of the time)
  • Schema Patch — modifies database table structure (add column, add index) — use only when db_schema.xml declarative schema can't do what you need
Say in interview

"Data Patches are how I install database data as part of a module — CMS blocks, configuration values, product attributes. They run once via setup:upgrade and are tracked in the database. The critical rule is never modify a patch after it's been deployed — you create a new one instead, because the old one is already recorded as 'done' in every environment."

Cloudflare — CDN, WAF, DDoS, SSL in one sentence each

The protective + performance layer in front of your store

▼

What Cloudflare Is

Cloudflare sits between the internet and your server. Every request goes through Cloudflare first. It can block bad traffic, cache content, and serve files faster — all before the request ever reaches your server.

User's browser ↓ [Cloudflare edge server — nearest to user] ✅ DDoS attack? Absorbed here. ✅ SQL injection attempt? Blocked by WAF. ✅ Static file? Served from cache here — fast. ✅ Dynamic page? Passed through to your server. ↓ [Your Magento server on GCP]

One Sentence per Feature

FeatureOne sentence
CDNCaches static files (images, CSS, JS) on servers near your users so they load fast from anywhere in the world.
WAF (Web App Firewall)Inspects every request and blocks known attack patterns — SQL injection, XSS, credential stuffing — before they reach Magento.
DDoS ProtectionAbsorbs massive floods of fake traffic (millions of requests/second) so your store stays online during attacks.
SSL/TLSFree HTTPS certificates with automatic renewal, and Cloudflare enforces HTTPS redirects so no customer ever browses on plain HTTP.
Bot ManagementIdentifies and blocks bots that scrape your prices, fake add-to-cart to deplete inventory, or try credential stuffing on customer accounts.
Page Rules / Cache RulesFine-tune what Cloudflare caches — e.g., cache category pages for 1 hour but never cache checkout pages.

Important for Magento: Bypass Cloudflare cache for cart, checkout, and customer-account pages. These are dynamic (personalized per user) — caching them would show the wrong data. Use page rules to set "Cache Level: Bypass" on /checkout/*, /customer/*, /cart.

Say in interview

"Cloudflare is a protective and performance layer that sits in front of the server. The CDN caches static files globally so they load fast everywhere. The WAF blocks attacks like SQL injection before they hit Magento. DDoS protection absorbs traffic floods. And SSL certificates are free and auto-renewed. For Magento, the key setup detail is making sure dynamic pages like checkout are excluded from caching."

Security basics — OWASP top issues + Magento fixes

The vulnerabilities that matter for e-commerce

▼

OWASP Top 10 — The E-Commerce Ones That Matter

VulnerabilityWhat it isHow Magento handles it
SQL InjectionAttacker injects SQL code via forms to read/destroy your DBMagento's ORM (models, collections, repositories) uses parameterized queries automatically. Never concatenate user input into raw SQL.
XSS (Cross-Site Scripting)Attacker injects malicious JS into pages that runs in other users' browsersAlways use $block->escapeHtml(), escapeUrl(), escapeJs() in .phtml templates. Magento blocks SVG uploads by default (SVGs can contain JS).
Broken AuthWeak passwords, no 2FA, session hijackingChange admin URL from /admin to custom path. Enable 2FA (Magento has built-in Google Authenticator). Strong password policy. Restrict admin to specific IPs.
CSRFTricking a logged-in user into submitting a malicious formMagento adds CSRF tokens to all forms automatically — verify your custom forms use the CSRF form key.
Sensitive Data ExposureCredit card data, passwords stored in plain textUse hosted payment gateways (Stripe, Paymob) — card data never touches your server. Magento hashes passwords with bcrypt.
Security MisconfigurationDebug mode on in production, default admin URL, error messages showing stack tracesDeploy in production mode (bin/magento deploy:mode:set production), hide error details, restrict file permissions.

Magento-Specific Security Checklist

  • ✅ Custom admin URL (e.g., /backend-xyz123 not /admin)
  • ✅ Two-Factor Authentication enabled for all admin users
  • ✅ Run in production mode — never developer mode on live
  • ✅ Apply security patches within days of release (subscribe to Adobe security alerts)
  • ✅ Correct file permissions: 664 for files, 775 for directories
  • ✅ No SVG upload without explicit security review
  • ✅ HTTPS enforced everywhere
  • ✅ Hosted payment gateway — card data never on your server
  • ✅ Regularly audit installed third-party extensions (they're attack vectors)

PCI DSS in Simple Terms

PCI DSS = Payment Card Industry Data Security Standard. Rules for anyone who handles credit card data. The easy way to comply: don't handle card data at all. Use a hosted/tokenized payment gateway where customers enter card details on the provider's secure page (iFrame or redirect). Your server only sees a token — never the actual card number.

GDPR in Simple Terms

GDPR = European data protection law. For an e-commerce store: show a cookie consent banner, allow customers to request their data be deleted, document what data you store and why, and don't send marketing emails without explicit opt-in.

Say in interview

"Security in Magento has a few layers. At the code level, I always use Magento's escaping methods in templates to prevent XSS, and the ORM for database queries to prevent SQL injection. At the platform level: custom admin URL, 2FA, production mode, and applying security patches quickly. For payments, I'd always use a hosted gateway so card data never touches our server — that alone dramatically reduces PCI DSS scope. And Cloudflare's WAF adds another layer by blocking known attack signatures at the edge."

REST vs GraphQL — when to use each

Two ways external systems talk to Magento

▼

What They Both Do

Both REST and GraphQL are APIs that let external systems (mobile apps, ERPs, dashboards, headless frontends) read and write Magento data over HTTP.

Side-by-Side

REST APIGraphQL API
URL structureMultiple endpoints: /rest/V1/products, /rest/V1/ordersSingle endpoint: /graphql
What you get backThe full object — even fields you don't needExactly the fields you asked for — nothing more
Multiple requestsNeed separate requests for related dataOne request can fetch nested related data
AuthenticationAdmin token or customer token in headerSame — token in Authorization header
Best forERP integrations, backend scripts, simple CRUDPWA/headless frontends, React/Vue apps, mobile
Your experienceWorking with middleware/ERP integrationsJolaUSA — Venia PWA consumed Magento's GraphQL

GraphQL Example — Get Only What You Need

# POST /graphql
{
  products(filter: { sku: { eq: "SKU001" } }) {
    items {
      name
      price_range {
        minimum_price {
          regular_price { value currency }
        }
      }
      thumbnail { url label }
    }
  }
}
# Response: only name, price, and thumbnail — nothing else
Say in interview

"Magento exposes both REST and GraphQL APIs. REST is great for server-to-server integrations like ERP or logistics systems — straightforward CRUD over specific endpoints. GraphQL is better for frontend applications because you request exactly the fields you need, reducing payload size. I've used both — I worked with Magento's GraphQL API at JolaUSA where I was building a React-based PWA using the Venia framework."

Why Asking Questions Matters

Good questions signal that you're thinking about the job seriously — not just trying to get hired, but figuring out if it's the right fit. It also shows senior developer thinking: you understand what makes projects succeed or fail.

Your 5 Questions (Ranked by Impact)

Question 1 — Platform status (critical to understand scope)

"What's the current state of the platform — are we starting completely from scratch with a new codebase, or is there existing code or infrastructure I'd be taking over?"

Question 2 — Third-party integrations (this is where complexity lives)

"What third-party systems will the platform need to integrate with — ERP, inventory management, logistics providers, payment gateways?"

Question 3 — Timeline (helps you understand priorities)

"What's the expected timeline for the first version to go live? And are there specific features that need to be there at launch?"

Question 4 — Communication structure (you'll be solo — this matters)

"Since this is a standalone developer role, who would I be coordinating with for requirements — directly with the business team, or is there a product manager or project coordinator in between?"

Question 5 — Infrastructure (to gauge how much ops work you'll own)

"Is GCP already provisioned, or would setting up the infrastructure be part of my responsibilities from day one?"

Don't ask about salary in the first interview (unless they bring it up). Save that for when they make an offer or ask you directly.

Questions to Avoid

  • "What does the company do?" — you should know this already
  • "What are the working hours?" — too early, sounds like you're checking the minimum
  • "Is there room for promotion?" — too soon, sounds presumptuous
